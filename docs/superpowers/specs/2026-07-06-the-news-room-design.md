# 《the news room》设计文档

> 2026-07-06 · v1 spec · 状态：待用户批准
> 一句话：一个只报道你的新闻世界——你的生活是这里唯一的公共事件。

## 0. 定位

表面是记录软件，内核是**私人媒体宇宙模拟器**：用户的人生是这个世界的唯一新闻源，围绕这个新闻源存在一个真实运转的媒体生态。产品不解决"我怎么记下来"，解决的是"**我为什么值得被记录**"。

灵魂句（产品文档级）：**让文本不是被生成出来，而是被一个世界生产出来。**

## 1. 产品原则（锁定）

1. **不把生成过程压缩成提示词。** 文章由媒体机构、编辑角色、竞争机制、职业压力和历史记忆共同驱动的世界内部运转产生，不是单次 prompt 的输出。中间过程本身是产品的第二内容面（后台可窥视），所以不存在"浪费的 token"。
2. **用户是唯一主角，但不是唯一声音。** 媒体可以赞美、误读、批评、质疑用户。用户可以回应、投诉、纠错、订阅、退订——市场权力是神级的；但用户没有编辑权，不能把世界降级成顺从的记录工具。**你是它们的神，不是它们的主编。**
3. **风格来自世界状态，不是下拉菜单。** 严禁"严肃/幽默/文艺"式的手动文风开关。文风只由机构立场、记者人格、员工状态、竞争压力、历史反馈涌现。换人=换整份履历记忆，文风变化必须是真的。
4. **允许夸大，但保留证据链。** 标题党和戏剧化合法（这是幽默感来源），但每篇文章区分**事实层**（引用的原始事件，可点击溯源）与**解释层**（媒体阐释）。用户永远知道哪句是事实、哪句是编辑部脑补。
5. **每次报道都沉淀回传记档案。** 任何一轮模拟结束必须回答三问：这件事在用户生活里意味着什么？和过去哪些主题有关？是否值得进入未来传记？防止世界变成"虚构报社跑团"而忘了主角。

## 2. 世界模型

### 2.1 机构（v1 三家 + 一个系统机构）

**「中央通讯社」**（系统机构，不拟人化）
入口管道：把用户原始输入改写成中性电头通稿 + 打 6 维评分，供各社抢稿。

**《用户日报》** — 严肃大报
- 编辑方针："严肃对待用户生活中的每一个结构性变化。"
- 风格：稳重、纪实、求证；负责主线叙事，独家承接敏感/沉重素材（伦理分工）。
- 岗位：总编辑**老周**（严厉，口头禅"这不是头条，这只是情绪噪音"）、主笔记者**沈望**（敏感文学化，缺点是过度阐释）、事实核查员**贺真**（冷静烦躁，讨厌小报，专杀"没有证据表明"）、毒舌专栏作家**秦刀**（方针"尊重用户，但绝不供奉用户"，负责指出自欺/拖延/工具癖——反彩虹屁的制度化保障）。

**《深夜头条》** — 娱乐小报
- 编辑方针："没有小事，只有不够炸裂的标题。"
- 风格：震惊体、抢流量、KPI 焦虑；最爱深夜行为、冲动消费、突然顿悟。
- 岗位：主编**麦浪**（流量焦虑，KPI 差时全社标题党程度上浮）、标题编辑**武震惊**（人如其名）、实习记者**周小满**（野心强，爆款后自信膨胀，常被打回）。

**《未来传记社》** — 长期传记机构
- 编辑方针："今天的一句话，可能是十年后某一章的开头。"
- 风格：命运感、章节感、回望式叙事；不追热点，只判断传记价值、埋伏笔。
- 岗位：传记主笔**林舟**（文学化，滥用"那一天后来被证明"，被贺真批评后会变克制）、档案管理员**顾卷**（沉默如图书馆幽灵，维护全世界的档案：人物/作品/地点/主题卡，重复模式检测由他触发）。

v2 扩编停车场：《精神地理》（文化杂志，把书/城市/工具写成精神地图）、《样本 C-258 研究所》（赛博学术机构，把用户当文明样本）、独立 newsletter 作者。

### 2.2 职员状态模型

每个职员 = **人格档案**（固定：性格、文风倾向、擅长题材、缺点、口头禅）+ **状态数值**（波动：心情、压力、自信、声望、野心）+ **履历记忆**（追加式档案：写过的每篇稿、每次被打回的理由、每次"当事人本人阅读了本文"的高光）+ **关系**（v1 最小化：与本社编辑的紧张度）。

写稿 prompt 的组装 = 人格档案 + 当前状态数值 + 履历记忆摘要 + 本次任务。**状态必须真实影响文本**：压力高→句子变短更急；被批评过→收敛；爆款后→更疯。

### 2.3 报社状态模型

每社：声望、财务（=订阅状态+用户阅读量折算）、标题党倾向、近期表现、总编辑严厉度。用户真实行为是唯一硬通货：点开阅读=当事人本人读了（记者最高荣誉，写进履历）、订阅/退订直接改财务、投诉触发更正流程。模拟大众读者只提供环境数字，权重远低于用户本人。

### 2.4 人事系统（v1 只留钩子，v2 启用）

离职/入职/转岗、人事变动公告（"林舟因长期过度阐释用户的工具购买行为，已被调往周末副刊"）、求职者带作品集应聘、每开一个新数据权限=编辑部发一张招聘启事。v1 数据结构预留 `agents.status`（active/departed），不实现流动逻辑。

## 3. 事件管线（每刊一轮）

```
用户输入(日记/采访回答/更正请求)
  → ① 通讯社发通稿：中性化改写 + 6维评分
       [新闻价值/传记价值/情绪强度/主题相关性/转折可能性/重复模式]
  → ② 顾卷归档：抽取人物/作品/地点/主题,关联历史事件,触发重复模式告警
       ("第20次焦虑时点奶茶" → 生成长期调查选题)
  → ③ 各社选题会：每社独立开会(多角色对话,产出选题单+派稿人),
       同一事件在三社命运不同(头版/边角料/冷处理)
  → ④ 写稿：被派职员按人格+状态+履历写稿(含标题)
  → ⑤ 审稿：《用户日报》走贺真事实核查(可打回一次,记者重写);
       《深夜头条》走武震惊改标题(夸大化)再由麦浪拍板;
       《未来传记社》走林舟自审+顾卷补档案引用
  → ⑥ 定版出刊：组装本期报纸(头版1+各版若干+编辑部内部日志1)
  → ⑦ 世界状态更新：职员数值增减、履历追加、报社声望/财务结算、
       档案沉淀三问(意义/关联主题/是否入传记) + 未来传记预览一句
```

- **刊期制**：每日两刊（早刊 07:30 / 晚刊 19:30，cron 触发）。截稿时间是真实机制：截稿前的输入进本期，截稿后进下期。世界不需要秒级运转——报纸本来就有刊期，**cron 即刊期，这是免费架构和叙事的双赢**。
- 无新输入的刊期照常出报（慢新闻日）：专栏、档案回顾（"一个月前的今天"）、采访约稿顶上——这本身是真实新闻业的样子。
- **采访即输入**：每刊由一名记者基于近期事件生成 1-3 个采访问题推给用户（名人访谈腔，不是心理咨询腔）。回答=下一期的独家专访素材；不回答积压 3 期后可出"多次联系当事人，均未获回应"（P2 彩蛋）。
- **更正机制**：用户对文章点"这不准"+一句说明 → 下期该报刊登《更正声明》，相关职员声望-1、履历记一笔。改日记变成体面的新闻流程。

## 4. 界面（两层）

**第一层 · 报纸前台**（好看、好刷、好截图）
- 首页=当期报纸：报头（the news room · 期号 · 日期）+ 头版大卡 + 分版瀑布流（快讯/深度/专栏/传记/档案回顾/专访）
- 文章页：正文 + 署名（记者卡片可点）+ 事实层引用（溯源到原始事件）+ 操作（订阅该报/这不准/让我上头条）
- 往期档案：按期翻旧报纸

**第二层 · 编辑部后台**（重度玩家的模拟经营面板）
- 今日选题会记录（多角色对话全文）
- 编辑部内部日志（"09:25 武震惊标题被贺真驳回"）
- 员工状态卡（数值+近况+履历高光）
- 报社声望/财务榜、被打回稿件墓地、标题竞争榜

**输入台**
- 写日记/随手记（textarea，一条=一个事件）
- 待答采访问题列表
- 新闻发布会按钮（P2：重大事件主动召开，全社连夜赶稿）

## 5. 数据模型（Netlify Blobs · 文档存储）

> 2026-07-06 修订：Supabase 出局（`*.supabase.co` 国内被墙实测 curl/代理均连接复位；haoqi 项目疑在 HK 区、pooler 也被掐；且无 access token 开新项目）。改用 Netlify Blobs——同站点存储，国内可达性已由 gaoyou/qiantaici 两个在用站点验证。

world store（单一 blob store）：
- `state/agencies.json` — [{id, name, charter, style, state:{声望,财务,标题党倾向,严厉度}, subscribed}]
- `state/agents.json` — [{id, agency_id, name, role, persona(人格档案), state:{心情,压力,自信,声望,野心}, status:active|departed}]
- `state/interviews.json` — [{id, asked_by, question, status:open|answered|expired, answer_event_id, edition_no}]
- `memories/<agentId>.json` — 履历记忆追加数组 [{kind:稿件|被打回|高光|批评|人事, content, edition_no, at}]
- `archive/cards.json` — 档案卡 [{kind:人物|作品|地点|主题, name, first_seen, mentions, bio_note}]
- `events/<ts>-<rand>.json` — 一切输入皆事件（append-only）：{id, type:diary|interview_answer|presser|read|flag|subscribe, raw_text?, article_id?, note?, question_id?, processed:false, wire_copy?, scores?, entities?, legacy_note?, created_at}。**用户的阅读/纠错/订阅行为也是事件**——由下一刊 tick 消费并结算世界状态，叙事与实现同构。
- `editions/index.json` — [{no, label, published_at, front_headline}]
- `editions/<no>.json` — 整期自包含：{no, label, published_at, front_article_id, articles:[{id, agency_id, author_agent_id, col, headline, body, fact_refs, facts:{event_id:raw_text 快照}, review_note, status}], meetings:[{agency_id, transcript}], backstage, world_log:[...]}（事实层快照内嵌，前端渲染零联表）

并发安全靠写者拓扑而非锁：`events/*` 只有前端追加（不可变），其余全部只有 tick 单写者。AI 生成内容（editions）与原始记录（events）分开存放（原则 4 的物理保障）。

## 6. 技术选型（多服务商免费拼接，无 Railway；2026-07-06 修订为 Netlify 主场）

| 部件 | 服务 | 免费额度依据 |
|---|---|---|
| 前端+API | Netlify 单站点：Next.js 静态导出（无 basePath）+ Functions（登录/读世界/写事件） | https://the-news-room-chichu.netlify.app ；国内可达性已由 gaoyou/qiantaici 验证；Functions 125k 次/月 |
| 世界状态 | Netlify Blobs（同站点，§5 布局） | 免费额度内，纯文本世界极小 |
| 刊期 tick | **GitHub Actions cron**（公有仓库标准 runner 免费，Node 脚本跑完整管线，经 token+siteID 外部读写 Blobs） | 每刊约 3-6 分钟 × 60 刊/月 |
| LLM | DeepSeek（日常全部岗位）+ StepFun step-3.7-flash（兜底，走 api.stepfun.ai，max_tokens≥1400）；Claude 留给未来年度特稿"明星主笔" | 每刊 20-30 次调用 ≈ 5-10 万 token/天，DeepSeek 价位≈每天几毛，一个月一杯奶茶 |
| 认证 | 单用户口令 → HMAC 会话 cookie；APP_PASSWORD_HASH + SESSION_SECRET 存站点环境变量 | 自带，零依赖 |

- Actions secrets：NETLIFY_AUTH_TOKEN、NETLIFY_SITE_ID、DEEPSEEK_API_KEY、STEPFUN_API_KEY。
- 部署走 `netlify deploy --prod`（qiantaici 同款，不吃 Netlify 构建分钟）。
- Supabase 出局原因记录于 §5 修订注——**别回头再选它，除非用户换了网络环境**。
- "催更/发布会"即时出刊（P2）：Function 持 GitHub PAT 调 workflow_dispatch。
- **模型分级=报社贫富**（设定钩子）：报社财务好→用更好的模型/更高 max_tokens，文风升级是涌现且叙事化的。v1 全员 DeepSeek，字段预留。
- 成本护栏：每刊调用上限（如 40 次）+ 单文章 token 上限。这不是省 token 哲学倒退，是防 bug 死循环刷爆账单的保险丝。

## 7. v1 范围

**做**（对应上面全部未标 P2/v2 的内容）：
1. 3 家报社 9 名职员，人格档案+状态数值+履历记忆全部真实落库
2. 完整事件管线（通稿→评分→归档→三社选题会→写稿→审稿打回→出刊→状态更新）
3. 每日早晚两刊 cron；慢新闻日自动填充
4. 输入台：日记 + 采访问答 + "这不准"更正
5. 前台报纸（首页/文章页/往期）+ 后台编辑部（选题会/内部日志/员工卡/榜单）
6. 未来传记预览（每个事件一句"未来传记可能这样写"）
7. 档案卡 v1-lite（主题/作品/人物计数 + 顾卷的重复模式检测）

**⚠️ 相对已批准方案的一处修正，待拍板**：原 v1 为"1 家报社 4 角色"，现改为"3 家报社 9 角色（每家更薄）"。理由：新闻社**互相竞争**是你亲手锁定的支柱，单报社版验证不了"同一事件在媒体市场中有不同命运"和标题竞争；GPT 稿的 MVP 也是 3 家。总调用量只多约一倍，仍在免费额度和奶茶预算内。**不同意就退回 1 家，管线不变。**

**不做（v2+ 停车场）**：人才市场/离职入职/人事公告、招聘启事=新权限、《精神地理》《样本研究所》、热搜榜、发布会即时刊、金铅字奖、年度人物、影响力档案馆（影响我最大的三本书——用户原始动机的正面回应，v2 优先）、传记章节生成、人生博物馆、L1/L2 数据权限（外卖/GitHub/Obsidian）、多用户化。

## 8. 风险与对策

| 风险 | 对策 |
|---|---|
| 后台戏太多，主角失位 | 原则 5 的沉淀三问是每刊管线的强制末步 |
| AI 彩虹屁日记，很快腻 | 秦刀专栏+贺真核查制度化毒舌；"62% 属于高级拖延"式的判断力是信任来源 |
| 敏感/沉重素材被小报消费 | 伦理分工：评分含情绪强度，高敏事件只派《用户日报》，小报自动回避 |
| 梗的保鲜 | 刊期节奏自调节：慢新闻日出专栏和档案回顾，不是每件事都炸裂 |
| 成本失控 | 每刊调用上限保险丝（见 §6） |
| 单 LLM 腔调穿透所有角色 | 履历记忆+状态数值进 prompt；验收标准见 §9，不达标就调整人格档案注入方式 |

## 9. 验收标准（v1 的"活着"判据）

1. **换人实验**：手动把沈望换成新记者（新人格+空履历），同类事件的报道文风肉眼可辨地不同。
2. **状态实验**：把周小满压力值拉满 vs 清零，各写一稿，输出有可感知差异。
3. **重复模式实验**：同主题事件输入 5+ 次，顾卷触发长期调查选题，稿内出现"这已是第 N 次"式的纵向引用。
4. **竞争实验**：同一事件，三社标题并排放，风格/立场/夸张度分明；至少出现过一次小报稿被核查打回的完整争吵记录。
5. 连续 3 天零输入，报纸照常出刊且内容不重复（慢新闻日机制有效）。
